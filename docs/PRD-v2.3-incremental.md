## 1. 变更概述

v2.3 是基于代码审查结论和用户决策的**定向增量变更**，不改变产品整体方向。核心策略：

- **做减法**：删除用户不需要的手动记录功能，明确 AI 识别为唯一面向用户的数据源
- **补短板**：添加系统托盘支持，让 Electron 应用符合桌面应用标准行为
- **优体验**：ReportPage 从渐进式向导改为单页全量配置，减少用户操作步数

## 2. 新增需求

### F-201: 系统托盘

#### 需求描述

当前应用关闭窗口时直接退出进程。应改为：**点击窗口关闭按钮（X）→ 窗口隐藏并最小化到系统托盘**，应用在后台继续运行（Tracker + Vision Auto 持续工作）。用户通过托盘图标唤回窗口或彻底退出。

#### 交互细节

| 场景 | 行为 |
|------|------|
| **点击窗口 X 按钮** | 窗口隐藏（`win.hide()`），不退出进程。托盘图标保持显示。 |
| **点击托盘图标** | 如果窗口已隐藏 → 显示并聚焦窗口；如果窗口可见 → 聚焦窗口 |
| **右键托盘菜单** | 弹出上下文菜单，包含两个选项：① **显示窗口** — 显示并聚焦主窗口；② **退出** — 关闭 Tracker/Vision Auto 后彻底退出应用 |
| **托盘图标** | 使用现有鸭子 logo SVG（`src/renderer/assets/xiabanya-logo.svg`），通过 `nativeImage` 创建 16×16 / 32×32 托盘图标 |
| **macOS Dock** | 保持现有行为：关闭窗口不退出（macOS 标准）；Dock 点击恢复窗口 |
| **Windows** | 关闭窗口 → 隐藏到托盘。托盘菜单「退出」才真正退出 |

#### 验收标准

- [ ] Windows: 点击窗口 X → 窗口隐藏，应用程序继续运行在系统托盘中
- [ ] Windows: 托盘图标可见（鸭子剪影），右键菜单包含「显示窗口」和「退出」
- [ ] Windows: 点击「显示窗口」→ 窗口恢复显示并聚焦
- [ ] Windows: 点击「退出」→ Tracker + Vision Auto 停止，数据库连接关闭，进程退出
- [ ] macOS: 点击窗口 X → 窗口关闭但不退出（Dock 图标保留），符合 macOS 标准行为
- [ ] macOS: 托盘图标正常显示，右键菜单功能正常
- [ ] 窗口隐藏期间，Tracker 和 Vision Auto 继续正常运行
- [ ] 托盘图标在应用启动时立即显示，不依赖窗口是否可见

#### 技术指引

- 主进程新建 `src/main/tray.ts`，导出 `createTray(window: BrowserWindow)` 函数
- 在 `app.whenReady()` 中创建窗口后调用 `createTray(mainWindow)`
- 使用 `BrowserWindow.on('close', ...)` 拦截关闭事件，调用 `event.preventDefault()` + `win.hide()`
- 托盘退出逻辑中调用 `app.exit(0)`（确保 `before-quit` 触发 → 数据库关闭）
- 参考 Electron 官方文档: [Tray](https://www.electronjs.org/docs/latest/api/tray)

## 3. 删除/降级需求

### 3.1 删除：RecordDialog 手动添加记录

**组件**: `src/renderer/components/RecordDialog.tsx`

**删除原因**:
- 该组件从未被任何页面或组件引用（grep 确认仅在自身文件中定义）
- 用户明确不需要手动添加/编辑追踪记录
- 产品定位：AI 自动识别是唯一数据入口，不需要人工补充

**操作**:
1. 删除 `RecordDialog.tsx` 文件
2. 检查并删除 preload/IPC 中仅服务于手动记录创建的相关通道（如 `records:create` 只服务于手动场景）—— 需代码审查确认

### 3.2 降级：activity_records 的产品角色

**现状**: 
- `useRecordsStore` 主要服务于 `TodayPage` 的 StatCard（"今日记录 N 条"）
- `activity_records` 表是 Tracker 自动写入的内部数据，AI 识别依赖它作为上下文

**v2.3 决策**:
- `vision_results` = 产品层面的**唯一面向用户的数据源**
- `activity_records` = **内部技术实现**，不再作为独立产品概念展示
- `TodayPage` StatCard 中"今日记录 N 条"可保留（作为技术指标展示），但应标注为"窗口追踪 N 条"而非独立的产品价值维度
- `useRecordsStore` 保留但降级为辅助数据源
- 不新增任何面向用户的 `activity_records` 查询/浏览页面

### 3.3 删除：PRD 中的过时条目

以下条目在旧版 PRD（v2.0 或更早）中存在但不再适用，v2.3 PRD 明确删除：

| 旧 ID | 描述 | 删除原因 |
|-------|------|----------|
| US-02 | 手动补充遗漏的活动 | 用户不需要手动记录 |
| F-002 | 活动手动记录功能 | 同上 |
| F-004 | 按日活动的数据表格 | 无面向用户的 activity_records 查询需求 |
| UI 4.4 | "活动记录页" | 该页面从未实现且不需要 |

## 4. 修改需求

### M-201: ReportPage — 渐进式向导 → 单页全量配置

#### 当前问题

ReportPage 使用 `step` 状态控制渐进式展示（0→1→2→3→4），每个步骤一个独立 Card：
- Step 0: 选择报告类型
- Step 1: 选择模板
- Step 2: 选择日期范围
- Step 3: 素材预览 + 生成按钮
- Step 4: 生成结果

**用户痛点**：无法一览所有配置，必须逐步点击才能看到后续步骤，修改上一项需要反复操作。

#### 修改方案：单页全量布局

所有配置区段在单个页面中一次性展示，垂直排列，无需步骤状态管理：

```
┌─────────────────────────────────────────────────────┐
│  生成报告                                            │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌─ 报告类型 ────────────────────────────────────┐  │
│  │  [日报] [周报] [月报]                          │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  ┌─ 模板选择 ────────────────────────────────────┐  │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────┐ │  │
│  │  │成果导向  │ │工作轨迹  │ │三句话    │ │TOP3│ │  │
│  │  │日报      │ │日报      │ │日报      │ │日报│ │  │
│  │  └──────────┘ └──────────┘ └──────────┘ └────┘ │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  ┌─ 日期范围 ────────────────────────────────────┐  │
│  │  [DateRangePicker: 开始日期] ~ [结束日期]      │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  ┌─ 素材预览（日期选择后自动加载）──────────────┐  │
│  │  🧠 AI 识别 42 条    🖥 窗口追踪 156 条       │  │
│  │  AI 截屏识别结果将作为报告主要内容源           │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │     [🧠 AI 生成报告（基于截屏识别+活动记录）]   │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
│  ── 生成结果（生成后出现在下方）─────────────────   │

│                                                     │
└─────────────────────────────────────────────────────┘
```

#### 交互变更

| 旧行为 | 新行为 |
|--------|--------|
| 选择报告类型 → 模板 Card 出现 | 所有配置始终可见 |
| 选择模板 → 日期 Card 出现 | 切换类型/模板即时高亮反馈 |
| 选择日期 → 素材预览 + 生成按钮出现 | 素材预览在日期选择后自动加载（异步），生成按钮始终可见但素材未加载时 disable |

| `step` 状态管理 5 个阶段 | 移除 `step`，用素材加载状态控制生成按钮可用性 |
| 生成结果替换配置区域 | 生成结果在配置区域下方追加显示，配置区域保留可修改重新生成 |

- [ ] 页面打开时，报告类型、模板、日期范围、素材预览、生成按钮**全部可见**（可能需要滚动）

## 6. 产品数据架构明确

### 核心原则

```
用户视角的数据流:

  桌面截屏 → AI 识别 (Vision) → vision_results → 产品展示
                                    ├── TodayPage (Hero Banner + 列表)
                                    ├── TimelinePage (时间线表格)
                                    ├── HeatmapPage (热力图)
                                    ├── AppsPage (应用统计)
                                    └── ReportPage (报告生成素材)

  窗口追踪 (Tracker) → activity_records → 内部技术实现
                                           ├── 为 Vision 识别提供上下文
                                           └── TodayPage StatCard 辅助指标
```

### 数据源定位

| 数据表 | 产品角色 | 用户可见 | 存储策略 |
|--------|----------|----------|----------|
| `vision_results` | **主要数据源** — AI 截屏识别结果 | ✅ 是 | 持久化，按月清理可选 |
| `activity_records` | **内部实现** — Tracker 窗口追踪记录 | ❌ 否（仅 StatCard 计数） | 持久化，为 Vision 提供上下文 |
| `reports` | **产出物** — 用户生成的报告 | ✅ 是 | 持久化 |
| `settings` | **配置** — 用户设置 | ✅ 是 | 持久化 |
| `screenshots` | **中间产物** — 截屏文件 | ❌ 否 | 默认不保留 |
